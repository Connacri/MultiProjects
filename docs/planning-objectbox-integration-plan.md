# ObjectBox Integration Plan

The existing hospital planning data is currently represented by the legacy `ActivitePersonne` DTO and the existing ObjectBox model. The DTO contains staff identity, group, team, month, schedule, observations and daily statuses, while the persisted ObjectBox entities must remain backward compatible. fileciteturn85file0L7-L36

The ObjectBox model already contains many unrelated application entities and generated model metadata. The migration must therefore be additive and must preserve existing ObjectBox UIDs. fileciteturn84file0L1-L5

## Migration strategy

1. Do not replace the existing `ActivitePersonne` DTO immediately.
2. Introduce new planning entities with new ObjectBox UIDs generated by `build_runner`.
3. Keep the legacy DTO as an adapter during migration.
4. Introduce a stable planning business key:
   - branchId
   - year
   - month
   - revision
5. Persist a `PlanningSnapshot` separately from staff master data.
6. Persist `RotationStateSnapshot` with the published snapshot.
7. Never derive future rotation continuity from mutable staff records if a persisted rotation state exists.
8. Migrate old monthly data into immutable snapshots only through an explicit import/migration command.
9. Never regenerate historical months as part of normal application startup or UI opening.

## Existing data compatibility

The legacy model uses textual month values such as `Octobre 2025` and stores daily statuses as a `List<String>`. The new model should normalize this into `(year, month, date)` records while preserving the original value during migration for traceability. fileciteturn85file0L7-L20

## ObjectBox rule

Never manually edit generated `objectbox.g.dart` identifiers or reuse an existing UID. Additive model changes must be generated by ObjectBox code generation and reviewed against `objectbox-model.json`.

## Publication transaction

The final publication transaction must persist:

- snapshot metadata;
- rotation state;
- all assignments;
- publication status.

All writes belong to one ObjectBox write transaction. If any write fails, the publication must not be visible as partially committed.
