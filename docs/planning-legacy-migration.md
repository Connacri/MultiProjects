# Legacy Planning Migration Strategy

## Current legacy model

The existing hospital planning data is stored around `Staff` and `ActiviteJour`.

`Staff` contains identity and team information (`equipe`), while `ActiviteJour` stores a day number and a status. This is suitable for the existing screen but does not contain a complete monthly planning identity, configuration version, or immutable snapshot metadata.

## Migration rule

Legacy data is treated as historical data.

The migration adapter is read-only:

```text
Legacy Staff + ActiviteJour
          ↓
LegacyPlanningReader
          ↓
PlanningSnapshot (historical)
```

The adapter never calls the rotation engine.

## Why this matters

If an old planning was saved with a manual adjustment, absence, or an old rotation rule, recalculating it with today's rules would corrupt historical data.

## ObjectBox safety

Do not change or manually reuse existing ObjectBox entity UIDs. The legacy entities remain untouched during the migration phase.

The new planning entities should receive new ObjectBox model IDs generated by `build_runner`. Existing model IDs must be preserved by ObjectBox model evolution.

## Legacy status mapping

The migration maps common statuses:

- `J`, `Jour`, `Day` → `day`
- `N`, `Nuit`, `Night` → `night`
- `R`, `RE`, `Repos`, `Rest` → `rest`

Unknown statuses are preserved as the assignment `code` and mapped to a neutral `other` shift type instead of being silently discarded.

## Important limitation

The legacy `ActiviteJour` entity stores only a day number. The migration therefore needs the month/year context supplied by the caller. The historical month must be explicitly identified before import.

## Recommended rollout

1. Keep legacy entities operational.
2. Add new planning entities with new ObjectBox UIDs.
3. Read old planning through `LegacyPlanningReader`.
4. Verify imported snapshots against the legacy grid.
5. Freeze the imported historical snapshot.
6. Generate only future unpublished months with the new engine.
7. Gradually migrate UI reads to the new repository.
8. Remove legacy writes only after all required historical data is migrated and verified.
